## Project Description

Dutch is a Splitwise-like expense-splitting Progressive Web App for groups of friends and flatmates. Users can create groups, add shared expenses, track who owes whom, and record settlements. The app is installable on Android, iOS, and desktop via PWA ‚Äî avoiding native app store distribution while delivering a native-app feel.

## Project

- **Name:** Dutch (codename: hisaab)
- **Type:** Progressive Web App (single Next.js application)
- **Purpose:** Track and settle shared expenses among friends and groups
- **Target Users:** Small groups (friends, flatmates, travel companions) splitting costs informally, primarily in India
- **Stage:** Prototype ‚Äî auth, expenses, friends, groups, activity feed, and settlements implemented

## Architecture Overview

Dutch is a single Next.js application using the App Router. There is no separate backend process ‚Äî server-side logic will live in API Routes and Server Actions within the same codebase.

```
Browser / PWA shell
       ‚îÇ
       ‚ñº
Next.js App Router (SSR + RSC)
       ‚îÇ
       ‚îú‚îÄ‚îÄ Server Components  ‚Üí  fetch data, render HTML
       ‚îú‚îÄ‚îÄ Server Actions     ‚Üí  mutations (form submit, settle up)
       ‚îú‚îÄ‚îÄ API Routes         ‚Üí  programmatic API (future mobile clients)
       ‚îÇ
       ‚ñº
Database (Neon ‚Äî serverless PostgreSQL via Drizzle ORM)
```

The service worker (generated by next-pwa / Workbox) caches static assets and page shells, enabling offline access to already-visited pages. Network-dependent data (balances, expenses) will degrade gracefully with skeleton/stale-UI patterns when offline.

## Auth Flow

**Mechanism:** Email OTP ‚Äî no passwords, no third-party OAuth.

```
[Any protected page]
        ‚îÇ no valid session
        ‚ñº
  /auth  ‚îÄ‚îÄ Step 1: Enter email ‚Üí POST /api/auth/otp/send
        ‚îÇ (OTP generated, SHA-256 hashed, stored; email sent via Resend)
        ‚ñº
  /auth  ‚îÄ‚îÄ Step 2: Enter 6-digit OTP ‚Üí POST /api/auth/otp/verify
        ‚îú‚îÄ‚îÄ existing user ‚Üí session created ‚Üí /dashboard
        ‚îî‚îÄ‚îÄ new user      ‚Üí session created ‚Üí /auth/setup

  /auth/setup ‚îÄ‚îÄ Enter name + username ‚Üí POST /api/auth/setup
        ‚ñº
  /dashboard  (is_onboarded = true)
```

**Session:** 32-byte CSPRNG hex token stored in `sessions` table. HTTP-only cookie `hisaab_session` (`Secure`, `SameSite=Lax`, 30-day Max-Age).

**Middleware (`src/middleware.ts`):** Runs on every non-public path. Validates session by joining `sessions` + `users` on Neon (Edge-compatible via `@neondatabase/serverless` WebSocket driver). Redirects unauthenticated users to `/auth`; redirects un-onboarded users to `/auth/setup`.

**OTP security:**
- 6-digit numeric, `crypto.getRandomValues`
- Stored as SHA-256 hash only
- Expires in 10 minutes
- Max 3 wrong attempts ‚Üí auto-invalidate
- Max 3 sends per email per hour ‚Üí 429

**Public routes (no session needed):** `/auth`, `/auth/setup`, `/api/auth/*`, `/_next/*`, static assets.

## Offline Architecture

Dutch is fully offline-capable for read and write.

**Read path (offline) ‚Äî Workbox NetworkFirst**
All `GET /api/*` responses (except `/api/auth/*`) are cached by Workbox with a 1-day TTL. Static JS/CSS/icon bundles are pre-cached. Any previously visited page loads from cache without a network connection.

**Write path (offline) ‚Äî IndexedDB mutation queue**
Every mutation goes through `useOfflineMutate()` (in `src/lib/offline/hooks.ts`):
- If `navigator.onLine` ‚Üí direct `fetch()`, returns `{ queued: false, response }`
- If offline ‚Üí stores a `PendingMutation` record in IndexedDB (`idb`), returns `{ queued: true }`; shows "Saved offline" toast; registers Background Sync tag

```
PendingMutation {
  id, url, method, body (JSON string), headers,
  label (human-readable, e.g. "Delete expense"),
  timestamp, status ("pending" | "error"), errorMessage?
}
```

**Sync path**
Triggered by: `window 'online'` event, app load (if online), or SW Background Sync postMessage.
`syncQueue()` replays mutations in timestamp order:
- 2xx ‚Üí `removeMutation(id)` ‚Üí dispatches `'dutch-data-refresh'` custom event
- non-2xx ‚Üí `markMutationError(id, serverMessage)` ‚Üí shown in SyncErrorDrawer
- network error ‚Üí stops loop (still offline)

**Background Sync (Chrome/Android only)**
`src/worker/index.ts` handles the `'dutch-sync'` SW sync event by postMessaging all open clients. The client calls `syncQueue()` on receiving the message. Falls back to `'online'` event on Safari/iOS.

**`'dutch-data-refresh'` event**
After any sync, `syncQueue()` dispatches this custom event on `window`. All pages that fetch data listen for it alongside the existing `'expense-added'` event to trigger a re-fetch.

**Limitations**
- Media uploads (presign + PUT to R2) are multi-step and cannot be atomically queued ‚Äî upload UI is disabled while offline.
- No optimistic UI for creates ‚Äî new items appear after sync completes. Pending count in OfflineBanner indicates queued state.
- Background Sync is Chrome/Edge only. OfflineBanner shows an iOS/Safari notice when the API is unavailable.

## Key Design Decisions

**1. Single Next.js app (no separate API server)**
Chosen to reduce operational complexity at prototype stage. API Routes and Server Actions live alongside the UI. Revisit if a mobile client needs a standalone API.

**2. next-pwa over manual service worker**
next-pwa (Workbox-based) handles asset caching strategies automatically. Trade-off: webpack-only, cannot use Turbopack. Mitigated by explicit `--webpack` flag in scripts.

**3. Monetary values as integer paise**
Floating-point arithmetic on currency causes rounding drift in split calculations. All amounts stored and computed as integers; display layer formats to ‚Çπ.

**4. Neon + Drizzle ORM**
Neon provides serverless PostgreSQL that scales to zero ‚Äî fits prototype economics. Drizzle is TypeScript-first with no magic and no code generation. `@neondatabase/serverless` supports both the Edge runtime (middleware, via WebSocket) and Node.js (API routes, via HTTP). Drizzle Kit manages schema migrations.

**5. shadcn/ui over a full component library**
Gives full control over component internals without a third-party design system's overrides. Components live in `src/components/ui/` and are never modified directly.

**6. Apple HIG as the design language**
All UI follows Apple Human Interface Guidelines ‚Äî frosted glass navigation (iOS `backdrop-filter: blur`), thin-weight typography (Geist variable font, weights 100‚Äì300 for chrome), generous rounded corners (`rounded-2xl`), hairline borders (`border-black/[0.06]`), and a single emerald accent colour. Rationale: the target audience is mobile-first (iOS/Android PWA install) and the Apple aesthetic is widely trusted for financial apps. Documented fully in the "Design Language ‚Äî Apple HIG" section of CLAUDE.md.

**7. Custom sessions over NextAuth/Lucia**
No third-party session library ‚Äî keeps the auth layer thin, fully transparent, and free of opinionated abstractions. Session tokens are 32-byte CSPRNG hex stored in a DB table; cookie is HTTP-only, Secure, SameSite=Lax. Trade-off: we own the expiry rolling, cleanup, and multi-device logic.
