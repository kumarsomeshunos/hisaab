## Project Description

Hisaab is a Splitwise-like expense-splitting Progressive Web App for groups of friends and flatmates. Users can create groups, add shared expenses, track who owes whom, and record settlements. The app is installable on Android, iOS, and desktop via PWA — avoiding native app store distribution while delivering a native-app feel.

## Architecture Overview

Hisaab is a single Next.js application using the App Router. There is no separate backend process — server-side logic will live in API Routes and Server Actions within the same codebase.

```
Browser / PWA shell
       │
       ▼
Next.js App Router (SSR + RSC)
       │
       ├── Server Components  →  fetch data, render HTML
       ├── Server Actions     →  mutations (form submit, settle up)
       ├── API Routes         →  programmatic API (future mobile clients)
       │
       ▼
Database (Neon — serverless PostgreSQL via Drizzle ORM)
```

The service worker (generated by next-pwa / Workbox) caches static assets and page shells, enabling offline access to already-visited pages. Network-dependent data (balances, expenses) will degrade gracefully with skeleton/stale-UI patterns when offline.

## Auth Flow

**Mechanism:** Email OTP — no passwords, no third-party OAuth.

```
[Any protected page]
        │ no valid session
        ▼
  /auth  ── Step 1: Enter email → POST /api/auth/otp/send
        │ (OTP generated, SHA-256 hashed, stored; email sent via Resend)
        ▼
  /auth  ── Step 2: Enter 6-digit OTP → POST /api/auth/otp/verify
        ├── existing user → session created → /dashboard
        └── new user      → session created → /auth/setup

  /auth/setup ── Enter name + username → POST /api/auth/setup
        ▼
  /dashboard  (is_onboarded = true)
```

**Session:** 32-byte CSPRNG hex token stored in `sessions` table. HTTP-only cookie `hisaab_session` (`Secure`, `SameSite=Lax`, 30-day Max-Age).

**Middleware (`src/middleware.ts`):** Runs on every non-public path. Validates session by joining `sessions` + `users` on Neon (Edge-compatible via `@neondatabase/serverless` WebSocket driver). Redirects unauthenticated users to `/auth`; redirects un-onboarded users to `/auth/setup`.

**OTP security:**
- 6-digit numeric, `crypto.getRandomValues`
- Stored as SHA-256 hash only
- Expires in 10 minutes
- Max 3 wrong attempts → auto-invalidate
- Max 3 sends per email per hour → 429

**Public routes (no session needed):** `/auth`, `/auth/setup`, `/api/auth/*`, `/_next/*`, static assets.

## Key Design Decisions

**1. Single Next.js app (no separate API server)**
Chosen to reduce operational complexity at prototype stage. API Routes and Server Actions live alongside the UI. Revisit if a mobile client needs a standalone API.

**2. next-pwa over manual service worker**
next-pwa (Workbox-based) handles asset caching strategies automatically. Trade-off: webpack-only, cannot use Turbopack. Mitigated by explicit `--webpack` flag in scripts.

**3. Monetary values as integer paise**
Floating-point arithmetic on currency causes rounding drift in split calculations. All amounts stored and computed as integers; display layer formats to ₹.

**4. Neon + Drizzle ORM**
Neon provides serverless PostgreSQL that scales to zero — fits prototype economics. Drizzle is TypeScript-first with no magic and no code generation. `@neondatabase/serverless` supports both the Edge runtime (middleware, via WebSocket) and Node.js (API routes, via HTTP). Drizzle Kit manages schema migrations.

**5. shadcn/ui over a full component library**
Gives full control over component internals without a third-party design system's overrides. Components live in `src/components/ui/` and are never modified directly.

**6. Apple HIG as the design language**
All UI follows Apple Human Interface Guidelines — frosted glass navigation (iOS `backdrop-filter: blur`), thin-weight typography (Geist variable font, weights 100–300 for chrome), generous rounded corners (`rounded-2xl`), hairline borders (`border-black/[0.06]`), and a single emerald accent colour. Rationale: the target audience is mobile-first (iOS/Android PWA install) and the Apple aesthetic is widely trusted for financial apps. Documented fully in the "Design Language — Apple HIG" section of CLAUDE.md.

**7. Custom sessions over NextAuth/Lucia**
No third-party session library — keeps the auth layer thin, fully transparent, and free of opinionated abstractions. Session tokens are 32-byte CSPRNG hex stored in a DB table; cookie is HTTP-only, Secure, SameSite=Lax. Trade-off: we own the expiry rolling, cleanup, and multi-device logic.
