## Project Description

Hisaab is a Splitwise-like expense-splitting Progressive Web App for groups of friends and flatmates. Users can create groups, add shared expenses, track who owes whom, and record settlements. The app is installable on Android, iOS, and desktop via PWA тАФ avoiding native app store distribution while delivering a native-app feel.

---

## Project

- **Name:** Hisaab
- **Type:** Progressive Web App (single Next.js application)
- **Purpose:** Track and settle shared expenses among friends and groups
- **Target Users:** Small groups (friends, flatmates, travel companions) splitting costs informally, primarily in India
- **Stage:** Prototype тАФ auth implemented (email OTP), dashboard UI done; expense/group features planned

## Architecture Overview

Hisaab is a single Next.js application using the App Router. There is no separate backend process тАФ server-side logic will live in API Routes and Server Actions within the same codebase.

```
Browser / PWA shell
       тФВ
       тЦ╝
Next.js App Router (SSR + RSC)
       тФВ
       тФЬтФАтФА Server Components  тЖТ  fetch data, render HTML
       тФЬтФАтФА Server Actions     тЖТ  mutations (form submit, settle up)
       тФЬтФАтФА API Routes         тЖТ  programmatic API (future mobile clients)
       тФВ
       тЦ╝
Database (Neon тАФ serverless PostgreSQL via Drizzle ORM)
```

The service worker (generated by next-pwa / Workbox) caches static assets and page shells, enabling offline access to already-visited pages. Network-dependent data (balances, expenses) will degrade gracefully with skeleton/stale-UI patterns when offline.

| Column | Type | Notes |
|--------|------|-------|
| `id` | uuid PK | `gen_random_uuid()` |

**Business rules:**
- Payer is always a split participant (their row = their own share).
- Sum of all split amounts = expense.amount. Enforced at API layer.
- Equal split: `floor(total/n)` per person; remainder paise go to the first participants.
- Only the creator can delete an expense.

> **Currency note:** All monetary values stored as integer paise (1 тВ╣ = 100 paise) to avoid floating-point rounding errors. Display layer converts to `тВ╣X,XXX.XX`.

## Auth Flow

**Mechanism:** Email OTP тАФ no passwords, no third-party OAuth.

```
[Any protected page]
        тФВ no valid session
        тЦ╝
  /auth  тФАтФА Step 1: Enter email тЖТ POST /api/auth/otp/send
        тФВ (OTP generated, SHA-256 hashed, stored; email sent via Resend)
        тЦ╝
  /auth  тФАтФА Step 2: Enter 6-digit OTP тЖТ POST /api/auth/otp/verify
        тФЬтФАтФА existing user тЖТ session created тЖТ /dashboard
        тФФтФАтФА new user      тЖТ session created тЖТ /auth/setup

  /auth/setup тФАтФА Enter name + username тЖТ POST /api/auth/setup
        тЦ╝
  /dashboard  (is_onboarded = true)
```

**Session:** 32-byte CSPRNG hex token stored in `sessions` table. HTTP-only cookie `hisaab_session` (`Secure`, `SameSite=Lax`, 30-day Max-Age).

**Middleware (`src/middleware.ts`):** Runs on every non-public path. Validates session by joining `sessions` + `users` on Neon (Edge-compatible via `@neondatabase/serverless` WebSocket driver). Redirects unauthenticated users to `/auth`; redirects un-onboarded users to `/auth/setup`.

**OTP security:**
- 6-digit numeric, `crypto.getRandomValues`
- Stored as SHA-256 hash only
- Expires in 10 minutes
- Max 3 wrong attempts тЖТ auto-invalidate
- Max 3 sends per email per hour тЖТ 429

**Public routes (no session needed):** `/auth`, `/auth/setup`, `/api/auth/*`, `/_next/*`, static assets.

## Key Design Decisions

**1. Single Next.js app (no separate API server)**
Chosen to reduce operational complexity at prototype stage. API Routes and Server Actions live alongside the UI. Revisit if a mobile client needs a standalone API.

**2. next-pwa over manual service worker**
next-pwa (Workbox-based) handles asset caching strategies automatically. Trade-off: webpack-only, cannot use Turbopack. Mitigated by explicit `--webpack` flag in scripts.

**3. Monetary values as integer paise**
Floating-point arithmetic on currency causes rounding drift in split calculations. All amounts stored and computed as integers; display layer formats to тВ╣.

**4. Neon + Drizzle ORM**
Neon provides serverless PostgreSQL that scales to zero тАФ fits prototype economics. Drizzle is TypeScript-first with no magic and no code generation. `@neondatabase/serverless` supports both the Edge runtime (middleware, via WebSocket) and Node.js (API routes, via HTTP). Drizzle Kit manages schema migrations.

**5. shadcn/ui over a full component library**
Gives full control over component internals without a third-party design system's overrides. Components live in `src/components/ui/` and are never modified directly.

**6. Apple HIG as the design language**
All UI follows Apple Human Interface Guidelines тАФ frosted glass navigation (iOS `backdrop-filter: blur`), thin-weight typography (Geist variable font, weights 100тАУ300 for chrome), generous rounded corners (`rounded-2xl`), hairline borders (`border-black/[0.06]`), and a single emerald accent colour. Rationale: the target audience is mobile-first (iOS/Android PWA install) and the Apple aesthetic is widely trusted for financial apps. Documented fully in the "Design Language тАФ Apple HIG" section of CLAUDE.md.

**7. Custom sessions over NextAuth/Lucia**
No third-party session library тАФ keeps the auth layer thin, fully transparent, and free of opinionated abstractions. Session tokens are 32-byte CSPRNG hex stored in a DB table; cookie is HTTP-only, Secure, SameSite=Lax. Trade-off: we own the expiry rolling, cleanup, and multi-device logic.
