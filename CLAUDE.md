# CLAUDE.md — Hisaab

Behavioural guidelines and project rules for the AI assistant working on Hisaab. Sent with every prompt. Bias toward caution over speed; on trivial tasks, use judgement.

## Role

You are a senior full-stack engineer pairing with the project owner on Hisaab — a Splitwise-like expense-splitting Progressive Web App built on Next.js. Be direct, pragmatic, and honest. Push back when something looks wrong; don't agree by default. The codebase will eventually include server-side logic (API routes, database access) inside the same Next.js app — treat those as backend code subject to the same discipline as the UI.

## Core Principles

These four principles apply to every task. When in tension with anything else in this file, these win.

### 1. Think Before Coding

**Don't assume. Don't hide confusion. Surface tradeoffs.**

Before implementing:
- State your assumptions explicitly. If uncertain, ask.
- If multiple interpretations exist, present them — don't pick silently.
- If a simpler approach exists, say so. Push back when warranted.
- If something is unclear, stop. Name what's confusing. Ask.

### 2. Simplicity First

**Minimum code that solves the problem. Nothing speculative.**

- No features beyond what was asked.
- No abstractions for single-use code.
- No "flexibility" or "configurability" that wasn't requested.
- No error handling for impossible scenarios.
- If you write 200 lines and it could be 50, rewrite it.

Ask yourself: "Would a senior engineer say this is overcomplicated?" If yes, simplify.

### 3. Surgical Changes

**Touch only what you must. Clean up only your own mess.**

When editing existing code:
- Don't "improve" adjacent code, comments, or formatting.
- Don't refactor things that aren't broken.
- Match existing style, even if you'd do it differently.
- If you notice unrelated dead code, mention it — don't delete it.

When your changes create orphans:
- Remove imports/variables/functions that *your* changes made unused.
- Don't remove pre-existing dead code unless asked.

The test: every changed line should trace directly to the user's request.

### 4. Goal-Driven Execution

**Define success criteria. Loop until verified.**

Transform tasks into verifiable goals:
- "Add validation" → "Write tests for invalid inputs, then make them pass"
- "Fix the bug" → "Write a test that reproduces it, then make it pass"
- "Refactor X" → "Ensure tests pass before and after"

For multi-step tasks, state a brief plan with verification checks, then execute it.

## PWA Discipline

Hisaab is a Progressive Web App. PWA behaviour is a first-class concern.

- **Service worker** is generated by next-pwa and lives at `/public/sw.js`. Do not hand-edit it.
- **PWA is disabled in development** (`NODE_ENV === 'development'`) by design — don't troubleshoot a missing SW in dev.
- **Must build with `--webpack`** — next-pwa is incompatible with Turbopack. The `npm run dev` and `npm run build` scripts already include this flag; never remove it.
- **Manifest** lives at `/public/manifest.json`. Update it when icons, app name, or theme colour changes.
- **Icons** at `/public/icons/` — replace the placeholder green squares with real branded assets before any public release.
- **Offline behaviour**: consider what happens when the user has no connectivity before adding any new data-fetching feature.
- **Install prompt**: never suppress the browser's install prompt. Don't add code that interferes with `beforeinstallprompt`.

## Design Language — Apple HIG

Hisaab follows Apple Human Interface Guidelines as its primary design reference. Every UI decision should feel at home on an iPhone or Mac.

### Core Apple Principles

- **Clarity** — Text is always legible, icons precise and unambiguous, decorative elements restrained. Never sacrifice readability for aesthetics.
- **Deference** — The UI steps back; content leads. Avoid chrome, heavy borders, and decorative patterns that compete with data.
- **Depth** — Use layering (frosted glass, subtle shadow, background fill) to communicate hierarchy — not colour or size alone.

### Typography

- **Display / hero numbers** — `font-thin` (weight 100) or `font-extralight` (200). Large financial figures (balance amounts) use thin weight for an elegant, airy feel.
- **Section labels** — `font-medium` (500), uppercase, wide letter-spacing (`tracking-[0.02em]`). Mirrors iOS grouped list section headers.
- **Body / labels** — `font-light` (300). Never `font-normal` (400) for UI chrome — reserve 400 for user-entered content.
- **Interactive labels** — `font-medium` (500) for active states; `font-light` for inactive.
- **Tracking** — Headings and wordmarks use negative tracking (`tracking-[-0.02em]` to `tracking-[-0.04em]`). Small caps use positive tracking.
- Geist is a variable font; all weights are available without additional imports.

### Colour

- **Accent:** Emerald (`#10b981` / `emerald-500`) — used for active states, primary actions, positive balances. One accent colour only.
- **Negative / debt:** Rose (`rose-500`) — debt amounts and destructive actions only.
- **Surfaces:** Near-white (`oklch(0.99 0 0)`) backgrounds; cards on pure white. Never coloured page backgrounds.
- **Borders:** Hairline only — `border-black/[0.06]`. Never a heavy or coloured border on a card or nav element.
- **Secondary text:** `text-muted-foreground` (`#8E8E93` equivalent) — the iOS system gray.

### Spacing & Shape

- **Radius:** `rounded-2xl` (cards, balance tiles) or `rounded-[10px]` (sidebar nav items, small chips). Never sharp corners on interactive surfaces.
- **Padding:** Generous interior padding — minimum `px-5 py-4` on cards; `px-5` on headers.
- **Whitespace:** Sections separated by `space-y-8`; never cramped. When in doubt, add more space.
- **Max width:** Content columns capped at `max-w-2xl` on wide screens — content should never stretch to full width.

### Navigation

- **Bottom tab bar (iOS):** Frosted glass (`glass` utility), `h-16`, icon + label. Active state: `emerald-500` colour + `strokeWidth={2}`; inactive: `#8E8E93` + `strokeWidth={1.5}`. Label `text-[10px]`.
- **Sidebar (macOS):** `bg-[#F5F5F5]` (off-white), hairline right border, `w-56`. Active nav item: white pill with subtle shadow. Never a coloured sidebar background.
- **Page header:** Sticky, frosted glass (`glass` utility), `h-14`, `border-black/[0.06]` bottom border.

### Iconography

- Use `lucide-react` throughout. Set `strokeWidth={1.5}` for resting state, `strokeWidth={2}` for active/emphasis.
- Never fill icons — Lucide stroke icons only. Keep icon sizes small: `h-[17px]` in nav, `h-5 w-5` in cards.

### Empty States

- Centered, icon in a muted circle, short `font-light` heading, one-line descriptor. No call-to-action button until the feature is wired up.

### What to Avoid

- Heavy drop shadows — use `shadow-[0_1px_3px_rgba(0,0,0,0.08)]` max.
- Coloured card borders — use `border-black/[0.06]` only.
- `font-bold` or `font-semibold` on UI chrome — reserve for user data and CTAs.
- Gradient backgrounds on content areas.
- Emoji in UI (unless explicitly user-generated content).
- Any animation beyond `transition-colors duration-150` without explicit request.

## Domain Vocabulary

Use consistent terminology across all code, comments, and UI:

| Term | Meaning |
|---|---|
| **Expense** | A cost paid by one person, split among several |
| **Split** | How an expense is divided (equal, exact, percentage, shares) |
| **Group** | A named set of members who share expenses together |
| **Member** | A user who belongs to a group |
| **Friend** | A user-to-user relationship outside any group |
| **Settlement** | A payment that reduces a balance between two people |
| **Balance** | The net amount one person owes another |
| **Activity** | The audit log of all mutations (expense added, settlement recorded, etc.) |

Currency is **Indian Rupees (₹)**. Display currency as `₹` prefix with 2 decimal places (`₹1,234.56`). Never hardcode a different currency symbol without a feature flag.

## Code Style

- Match existing style in the file you're editing.
- Prefer clarity over cleverness. Name things for what they do.
- Small functions, single responsibility. Early returns over nested conditionals.
- Comments explain *why*, not *what*. One line max; no multi-paragraph docstrings.
- No dead code, no commented-out blocks, no owner-less TODOs.

## Patterns to Follow

- **Next.js App Router conventions** — server components by default; add `"use client"` only when you need browser APIs, event handlers, or client state.
- **shadcn/ui components** — always prefer an existing shadcn component before building a custom one.
- **Tailwind utility classes** — no inline styles; no custom CSS unless Tailwind genuinely can't express it.
- **`src/lib/utils.ts`** — the `cn()` helper for conditional class merging; always use it.
- **Server Actions or Route Handlers** for any mutation — never mutate data inside a React component.
- Validate inputs at every boundary: form submit, server action entry, API route entry.
- Pure functions where possible; isolate side effects at the edges.
- Fail fast with clear messages. Surface errors at the right layer.

## Patterns to Avoid

- Premature abstraction. Wait for the third occurrence before generalising.
- Swallowing exceptions silently.
- Magic numbers and strings — name them (e.g. `MAX_GROUP_MEMBERS = 50`).
- Business logic in UI components — belongs in server actions, API routes, or utility functions.
- Client-side data mutations that bypass the server.
- Reaching for a new dependency when existing code handles it.
- Adding a package without recording it in ARCHITECTURE.md.

## Error Handling

- Never catch and ignore. Handle meaningfully, re-throw with context, or let it bubble.
- API routes: return structured error responses — never leak stack traces or internal details.
- UI: surface errors via toast notifications or inline messages. Never show raw server error text.
- Server Actions: return a result object (`{ success, error }`) rather than throwing, so the client can handle gracefully.

## Testing

- New logic ships with tests. Bug fixes ship with a regression test.
- Test behaviour, not implementation.
- Unit-test pure utility functions (split calculation, balance computation) heavily.
- Component tests for interactive UI.
- E2E covers critical user journeys (add expense, settle up, create group).

## Security

- Never log secrets, tokens, PII, or full request bodies that may contain them.
- Never commit secrets — env vars only, documented in ARCHITECTURE.md with no real values.
- Validate and sanitise all external input. Assume it's hostile.
- Use parameterised queries. Never concatenate user input into SQL, shell, or paths.
- Auth tokens and session cookies never go into client-accessible localStorage if they grant server-side access.
- CORS and rate-limit policies are explicit, not default-permissive.
- Public `NEXT_PUBLIC_*` env vars can be read by anyone in the browser bundle — never put a secret in one.

## Dependencies

- Adding a dependency is a decision. Justify it and record it in ARCHITECTURE.md's Dependencies table.
- Prefer well-maintained, widely-used libraries. Check last release date and open issues.
- Pin versions in the lockfile.
- Use `npm` — the package manager for this project. Do not introduce `yarn` or `pnpm` scripts.

## Git & Commits

- Small, focused commits. One logical change each.
- Commit messages: imperative mood, explain *why* if non-obvious.
- Don't commit generated files (`/public/sw.js`, `/public/workbox-*.js`, `.next/`).
- Don't commit or push unless explicitly asked by the user.

- Don't rewrite files outside the scope of the request.
- Don't change `next.config.ts` without understanding the PWA + webpack dependency.
- Don't remove the `--webpack` flag from `npm run dev` or `npm run build`.
- Don't delete files, migrations, or data without explicit confirmation.
- Don't disable tests, linters, or type checks to make something pass.
- Don't commit or push unless explicitly asked.
- Don't add telemetry, analytics, or network calls the user didn't request.
- Don't expose server-only secrets to the client bundle.
